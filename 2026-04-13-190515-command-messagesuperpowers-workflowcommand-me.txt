⏺ 方案对比

  方案 A: 独立 Schedule 模块（推荐）

  modules/
  ├── schedule/              # 新模块
  │   ├── entities/          # Job + JobExecution 实体
  │   ├── schedule.service.ts   # 调度核心（CRUD + 运行时 + 超时/重试）
  │   ├── job-agent.service.ts  # 执行 Agent（独立 LangGraph 图）
  │   ├── schedule.controller.ts
  │   └── schedule.module.ts
  ├── langgraph/
  │   └── tools/
  │       └── collections/
  │           └── schedule.tools.ts  # cron_job 工具（加入现有 loader）

  优点：模块边界清晰，与现有架构一致（参考 AgentModule、RuleModule 模式），执行
  Agent 通过 ToolRegistryService 获取工具（零耦合）

  缺点：需要处理 ScheduleModule ↔ LangGraphModule 的循环依赖（用 forwardRef）

  方案 B: 嵌入 LangGraph 模块

  把 JobService 放进 langgraph/ 模块内，执行 Agent 复用主图的 agent node。

  优点：无循环依赖，文件少

  缺点：违反单一职责，langgraph 模块已经很复杂（双图模式 +
  supervisor），职责耦合

  方案 C: 事件驱动

  Schedule 模块通过 NestJS EventEmitter 发射 job.triggered 事件，LangGraph
  模块监听执行。

  优点：完全解耦

  缺点：过度工程化，单进程场景下事件总线增加调试复杂度，没有实际收益

  推荐方案 A。理由：与现有模块组织模式一致（AgentModule、RuleModule 都是独立模块
   + 工具注册），执行 Agent 通过注入 ToolRegistryService 获取工具，不直接依赖
  LangGraph 模块。

⏺ 第 3 节：核心流程

  流程 1：用户通过聊天创建定时任务

  用户: "每天早上9点给我发天气预报邮件"
    → 现有 LangGraph 图（对话 Agent）
    → LLM 调用 cron_job 工具: { action: "add", type: "cron", cron: "0 9 * * *",
  instruction: "查询今天的天气预报并发送邮件给用户" }
    → cron_job 工具 → ScheduleService.addJob()
    → ScheduleService: 保存 Job 到 DB + 注册到 SchedulerRegistry
    → 返回结果给 LLM → 回复用户

  流程 2：定时任务触发执行

  SchedulerRegistry 定时触发
    → ScheduleService.executeJob(job)
    → 并发保护: runningJobs Set 检查
    → 创建 JobExecution 记录 (status=running)
    → JobAgentService.runJob(instruction)
      → 构建独立 LangGraph 图（绑定全部工具，不含 cron_job）
      → SystemMessage("你是后台任务执行代理...") + HumanMessage(instruction)
      → while 循环: invoke → tool_calls? → 执行工具 → 回传结果 → 再 invoke
      → 返回最终文本结果
    → 超时保护: Promise.race([runJob, timeout])
    → 成功: 更新 JobExecution (success) + 重置 consecutiveFailures
    → 失败: 更新 JobExecution (failed) + 指数退避重试 + 累计 consecutiveFailures
    → 连续失败 >= autoDisableThreshold: 自动停用 + stopRuntime

  流程 3：应用启动恢复

  onApplicationBootstrap():
    → 查询 DB: isEnabled=true 的 Jobs
    → 去重检查: SchedulerRegistry 是否已注册
    → startRuntime(job) 注册每个任务
